## 规则 1：测试数据真实性规范

> **核心原则**：测试数据必须看起来像真实用户发送的内容，避免使用明显的测试标识符，从而减少测试数据清理工作。

### 触发场景

| 场景 | 必须遵守 |
|------|---------|
| 创建测试用户数据 | ✅ 强制 |
| 编写 Mock 数据 | ✅ 强制 |
| 添加测试消息/内容 | ✅ 强制 |
| 数据库种子数据 | ✅ 强制 |

### 真实性要求

**✅ 正确做法**：
- 用户名：使用真实的中文名或英文名（如「张伟」「Alice」）
- 消息内容：使用有意义的句子，而非 `test123`、`测试测试`
- 时间戳：使用合理的日期范围，而非 `1970-01-01`
- 头像：使用真人风格的图片 URL 或 placeholder
- 邮箱：使用看起来真实的邮箱格式（如 `zhangwei@example.com`）

**❌ 禁止做法**：
- 用户名：`test_user_1`、`测试账号`、`MockUser`
- 消息内容：`test`、`hello world`、`这是一条测试消息`
- 时间戳：`1970-01-01 00:00:00`、`9999-12-31`
- 明显的测试标识：`test_`、`mock_`、`demo_` 前缀

### 示例对比

```javascript
// ❌ 错误：明显的测试数据
const mockUser = {
  name: 'test_user_1',
  email: 'test@test.com',
  message: '这是一条测试消息'
};

// ✅ 正确：看起来像真实用户
const mockUser = {
  name: '张伟',
  email: 'zhangwei@life.com',
  message: '今天的天气真不错，想去公园走走'
};
```

### 好处

| 好处 | 说明 |
|------|------|
| **无需清理** | 测试数据与真实数据混在一起，不影响用户体验 |
| **演示友好** | 产品演示时无需切换环境或隐藏测试数据 |
| **视觉一致** | 开发时看到的界面与生产环境一致 |
| **减少维护** | 不需要维护独立的测试数据清理脚本 |

### 注意事项

| 注意点 | 说明 |
|--------|------|
| 数据量控制 | 即使真实，测试数据量也不应过多（不超过生产数据的 10%） |
| 可识别性 | 内部可通过隐藏字段（如 `is_test: true`）标记，但对外不可见 |
| 隐私合规 | 使用虚构的真实风格数据，不使用真实个人信息 |

## 规则 2：合并冲突人工介入协议

> **核心原则**：AI 遇到 merge conflict 时必须暂停，严禁自行决策覆盖代码，必须人工介入确认。
> 
> **事故案例**：小心愿详情页底部导航栏在合并时被 AI 自行覆盖消失，导致功能丢失。
> 
> **全局规则**：此规则同时存在于全局 `~/.config/opencode/AGENTS.md`（规则 9），适用于所有项目。

### 触发场景

| 场景 | 必须人工介入 |
|------|-------------|
| `git merge` 产生冲突 | ✅ 强制 |
| `git rebase` 产生冲突 | ✅ 强制 |
| `git pull` 产生冲突 | ✅ 强制 |
| GitHub PR 合并冲突 | ✅ 强制 |
| 分支合并/同步操作 | ✅ 强制 |

### 🚫 禁止行为（FATAL ERROR）

| 禁止操作 | 原因 |
|----------|------|
| 自行选择 `ours` 或 `theirs` | 不理解两边代码的功能含义，可能丢失功能 |
| 简单拼接两边代码 | 可能产生逻辑冲突或重复代码 |
| 假设"新代码一定更好" | 新代码可能只是部分功能，缺少旧代码的完整功能 |
| 不展示冲突内容就解决 | 用户无法知晓被覆盖了什么 |
| 跳过询问直接合并 | 用户丧失决策权 |

### ✅ 强制执行流程

**步骤 1：暂停并报告**
```
⚠️ 检测到合并冲突，需要人工介入。

冲突文件：{文件路径}
冲突位置：{行号范围}
```

**步骤 2：展示两边代码及其功能含义**

必须向用户清晰展示：
```markdown
## 冲突分析

### 📍 冲突位置
- 文件：`src/pages/WishDetail.vue`
- 行号：120-150

### 🔴 当前分支代码 (ours)
```代码内容```
**功能含义**：[描述这段代码实现了什么功能]

### 🔵 合入分支代码 (theirs)
```代码内容```
**功能含义**：[描述这段代码实现了什么功能]

### ⚠️ 功能差异分析
- ours 包含：[列出功能点]
- theirs 包含：[列出功能点]
- 独有功能（ours）：[仅 ours 有的功能]
- 独有功能（theirs）：[仅 theirs 有的功能]
```

**步骤 3：询问用户决策**

```
请选择合并策略：
1. 保留 ours（当前分支） — 保留 {功能点列表}
2. 保留 theirs（合入分支） — 保留 {功能点列表}
3. 手动合并（需要合并两边功能） — 请说明如何合并
4. 查看更多上下文 — 展示更多代码帮助理解

请输入选项编号或直接描述你的合并意图。
```

**步骤 4：用户确认后执行**

只有当用户**明确回复**决策后，AI 才能执行合并操作。

### 功能丢失风险识别

AI 在展示冲突时，必须主动检查并高亮：

| 风险信号 | 说明 |
|----------|------|
| 一边有 UI 组件，另一边没有 | 可能丢失页面元素 |
| 一边有事件处理函数，另一边没有 | 可能丢失交互功能 |
| 一边有配置项/属性，另一边没有 | 可能丢失配置能力 |
| 一边有 import，另一边没有 | 可能丢失依赖 |

如果发现以上风险信号，必须在询问时**额外提醒**：
```
⚠️ 风险提示：选择 theirs 将丢失 [底部导航栏组件]，该功能在 ours 中存在但 theirs 中缺失。
```

### 重要约束

| 约束 | 说明 |
|------|------|
| 🔴 **必须暂停** | 发现冲突立即停止，不能继续执行 |
| 🔴 **必须展示两边** | 展示完整冲突代码，不能省略 |
| 🔴 **必须分析功能** | 描述两边代码各自实现了什么功能 |
| 🔴 **必须询问用户** | 等待用户明确回复后才能执行 |
| 🔴 **必须风险提示** | 发现功能丢失风险时主动提醒 |
